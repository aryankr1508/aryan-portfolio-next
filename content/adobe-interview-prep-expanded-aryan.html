4. Spring Boot from absolute beginner to interview level

The minimum Spring Boot depth needed to defend a Java backend role.

One-line mental model

Spring Boot is a Java backend machine where HTTP requests enter controllers, business logic runs in services, database calls happen through repositories, and JSON responses go back to the frontend.

Interview posture

You do not need to sound like a 10-year Spring architect. You need to explain API flow, annotations, transactions, validation, security, and testing confidently.

Basic file/folder API example

POST /api/folders
Request Body:
{
  "name": "Invoices",
  "parentFolderId": 10
}

Backend flow:
1. JWT filter authenticates user
2. Controller receives request
3. @Valid validates DTO
4. Service checks permission on parent folder
5. Repository saves folder entity
6. Service writes audit log
7. Controller returns FolderResponse with 201 Created

Spring Boot Q&A

What is Spring Boot in simple words?

Spring Boot is a framework that makes it faster to build Java applications, especially REST APIs and microservices. It sits on top of Spring and gives auto-configuration, embedded server, starter dependencies, production-ready defaults, and easy integration with databases/security/testing.

Interview version: Spring Boot reduces boilerplate and lets us build production-ready Java services quickly. For example, adding spring-boot-starter-web gives Tomcat, DispatcherServlet, JSON serialization, and REST controller support automatically.

Say: Controller should stay thin. Business logic belongs in service. Repository should only handle persistence. DTO protects API contract from internal DB entity changes.

What is Dependency Injection?

Dependency Injection means an object does not create its dependencies manually. Spring creates and injects them. This makes code loosely coupled and testable.

public FileController(FileService fileService) {
    this.fileService = fileService;
}

Here, the controller does not say new FileService(). Spring injects the service.

@Component vs @Service vs @Repository?
  • @Component: generic Spring-managed bean.
  • @Service: business logic class. Semantically clearer than Component.
  • @Repository: persistence layer. Also helps translate DB exceptions into Spring DataAccessException hierarchy.
@RestController vs @Controller?

@Controller is used for MVC pages/views. @RestController = @Controller + @ResponseBody; it returns JSON/XML directly. For REST APIs, use @RestController.

What is DTO and why not return Entity directly?

DTO means Data Transfer Object. It represents API input/output. Entity represents database table. We avoid returning entities because entities may expose internal fields, create circular JSON, tightly couple API to DB schema, and make future changes harder.

What is @Transactional?

@Transactional runs a method inside a database transaction. If all operations succeed, commit. If runtime exception occurs, rollback.

@Transactional
public void moveFile(Long fileId, Long targetFolderId) {
    File file = fileRepository.findById(fileId).orElseThrow();
    file.setFolderId(targetFolderId);
    auditRepository.save(new AuditLog("FILE_MOVED", fileId));
}

If audit save fails, file move also rolls back.

What is @ControllerAdvice?

It is used for global exception handling. Instead of writing try/catch in every controller, define one global handler that converts exceptions into proper HTTP responses.

@ControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ApiError> handleNotFound(ResourceNotFoundException ex) {
        return ResponseEntity.status(404).body(new ApiError("NOT_FOUND", ex.getMessage()));
    }
}
What is validation in Spring Boot?

Validation checks incoming DTO fields before business logic runs. Use annotations like @NotBlank, @NotNull, @Size, and trigger them with @Valid.

public class CreateFolderRequest {
    @NotBlank
    private String name;

    @NotNull
    private Long parentFolderId;
}

@PostMapping("/folders")
public FolderResponse create(@Valid @RequestBody CreateFolderRequest request) {
    return folderService.create(request);
}
What is Spring Data JPA?

Spring Data JPA simplifies database access. Instead of manually writing SQL for common CRUD operations, we extend JpaRepository. It generates methods like save, findById, findAll, deleteById.

public interface FileRepository extends JpaRepository<FileEntity, Long> {
    List<FileEntity> findByFolderIdAndDeletedFalse(Long folderId);
}
What is the N+1 query problem?

Example: You fetch 100 files, then for each file Hibernate separately fetches owner/user. That becomes 1 + 100 queries. Fix with fetch joins, EntityGraph, batch fetching, or DTO queries.

@Query("select f from FileEntity f join fetch f.owner where f.folderId = :folderId")
List<FileEntity> findFilesWithOwner(Long folderId);

Controller-Service-Repository code skeleton

@RestController
@RequestMapping("/api/files")
public class FileController {
    private final FileService fileService;

    public FileController(FileService fileService) {
        this.fileService = fileService;
    }

    @PostMapping
    public ResponseEntity<FileResponse> createFile(@Valid @RequestBody CreateFileRequest request) {
        FileResponse response = fileService.createFile(request);
        return ResponseEntity.status(HttpStatus.CREATED).body(response);
    }
}

@Service
public class FileService {
    private final FileRepository fileRepository;

    public FileService(FileRepository fileRepository) {
        this.fileRepository = fileRepository;
    }

    @Transactional
    public FileResponse createFile(CreateFileRequest request) {
        // 1. Validate business rule
        // 2. Check permission
        // 3. Save DB entity
        // 4. Return DTO
        FileEntity entity = new FileEntity();
        entity.setName(request.getName());
        entity.setFolderId(request.getFolderId());
        FileEntity saved = fileRepository.save(entity);
        return new FileResponse(saved.getId(), saved.getName());
    }
}

public interface FileRepository extends JpaRepository<FileEntity, Long> {
    List<FileEntity> findByFolderIdAndDeletedFalse(Long folderId);
}

5. Java core interview notes

Java fundamentals Adobe can ask before or after project discussion.

What to master first

Explain OOP pillars with examples.
  • Encapsulation: hide internal data through private fields and public methods.
  • Abstraction: expose what object does, hide how it does it. Example: Payment interface.
  • Inheritance: child class reuses parent behavior.
  • Polymorphism: same method call behaves differently depending on object.
interface StorageProvider {
    void upload(File file);
}

class GoogleDriveProvider implements StorageProvider {
    public void upload(File file) { /* Google API */ }
}

class OneDriveProvider implements StorageProvider {
    public void upload(File file) { /* Microsoft API */ }
}
Interface vs abstract class?

Interface defines contract/capability. A class can implement multiple interfaces. Abstract class provides partial implementation and shared state. Use interface when unrelated classes share behavior contract; abstract class when classes are closely related and share base logic.

HashMap internal working.

HashMap calculates key hashCode, converts it into bucket index, stores key-value entry. If collision occurs, entries are stored in linked list; after threshold, Java 8+ treeifies bucket into red-black tree. During get, it finds bucket using hash, then compares keys using equals.

  • Average get/put: O(1)
  • Worst-case before Java 8: O(n)
  • Worst-case after treeification: O(log n)
HashMap vs ConcurrentHashMap.

HashMap is not thread-safe. ConcurrentHashMap is thread-safe and allows concurrent reads/writes with better performance than synchronizing whole map. Use ConcurrentHashMap when multiple threads update shared map.

ArrayList vs LinkedList.
ArrayListLinkedList
Backed by dynamic arrayBacked by nodes
Fast random access O(1)Slow random access O(n)
Insertion in middle costlyInsertion easier if node known
Better default choice in most casesUseful for queue/deque-like operations but ArrayDeque often preferred
Checked vs unchecked exceptions.

Checked exceptions must be handled or declared. Example: IOException. Unchecked exceptions extend RuntimeException and are not compiler-enforced. Example: NullPointerException, IllegalArgumentException.

Interview rule: checked for recoverable external failures; unchecked for programming errors or invalid business state.

What is final?
  • final variable: cannot reassign.
  • final method: cannot override.
  • final class: cannot extend.
What is volatile?

volatile ensures visibility across threads. Reads/writes go to main memory, not stale CPU cache. It does not make compound operations atomic. For counters, use AtomicInteger or synchronized.

synchronized vs Lock?

synchronized is built into Java and locks block/method. Lock gives more control: tryLock, timed lock, interruptible lock, fairness. For simple cases synchronized is enough; for advanced lock management use ReentrantLock.

What are Java 21 virtual threads?

Virtual threads are lightweight threads introduced through Project Loom. They are useful for I/O-heavy tasks because many virtual threads can share a smaller number of platform threads. They simplify concurrent code compared to reactive style for many blocking I/O workflows.

Safe answer: I understand them as useful for high-concurrency I/O-bound workloads, such as reading multiple sensor/file/API streams concurrently, while still writing straightforward blocking-style code.

equals() and hashCode() contract.

If two objects are equal according to equals, they must have same hashCode. If they have same hashCode, they may or may not be equal. Hash-based collections depend on this contract.

String vs StringBuilder vs StringBuffer.
  • String is immutable.
  • StringBuilder is mutable and not thread-safe, fastest for local string construction.
  • StringBuffer is mutable and synchronized, thread-safe but slower.
Streams vs loops.

Streams provide declarative data processing: filter, map, collect. They are readable for transformations but not always faster. Loops are clearer for complex control flow, early break, or performance-critical code.

Beginner-to-advanced answer pattern

First define the concept simply. Then explain internal working. Then give one project example. Then mention trade-off. Example: “Redis cache improves speed, but stale data risk means we need TTL and invalidation.”

Q&A

GET vs POST vs PUT vs PATCH vs DELETE?
  • GET fetches data and should not change server state.
  • POST creates or triggers non-idempotent operation.
  • PUT replaces full resource and is idempotent.
  • PATCH partially updates resource.
  • DELETE removes resource and is idempotent.
What is idempotency?

An operation is idempotent if repeating it multiple times has same final result. GET, PUT, DELETE are generally idempotent. POST is not usually idempotent unless using an idempotency key.

Project example: Move file should be idempotent if moving same file to same target folder repeatedly leaves the system in same state.

401 vs 403?

401 means user is not authenticated or token is invalid. 403 means user is authenticated but lacks permission.

Example: no JWT → 401. Valid JWT but no access to folder → 403.

What is JWT?

JWT is a signed token carrying claims like userId, roles, expiry. Backend verifies signature and expiry on every protected request. It avoids DB lookup for basic authentication but authorization may still need fresh permission data.

What should be inside JWT?

Minimal claims: userId, email/username, roles/scopes, issuedAt, expiry, token id. Avoid sensitive data like password, secrets, PII-heavy info.

Where to store JWT?

Options: HTTP-only secure cookie or memory/storage depending on architecture. HTTP-only cookie helps protect against XSS token theft but needs CSRF mitigation. Authorization header is common for APIs but frontend storage must be handled carefully.

Authentication vs Authorization?

Authentication asks “who are you?” Authorization asks “what are you allowed to do?” Login/JWT validation is authentication. Checking whether user can delete a folder is authorization.

What is OAuth?

OAuth is an authorization framework that lets a user grant an app limited access to another service without sharing password. Example: user connects Google Drive; app receives access token with specific scopes.

OAuth vs JWT?

OAuth is a protocol/framework for delegated authorization. JWT is a token format. OAuth can use JWT as access token, but they are not the same thing.

What is RBAC?

Role-Based Access Control assigns permissions through roles. Example: Owner, Admin, Editor, Viewer. User gets roles; roles contain permissions; APIs check required permission.

What is CORS?

CORS is browser security controlling whether frontend from one origin can call backend on another origin. Backend must allow origins, methods, and headers. It is not authentication.

What is CSRF?

CSRF tricks authenticated browser into sending unwanted request using cookies. Mitigate with SameSite cookies, CSRF tokens, and checking origin/referer for sensitive operations.

How do you design secure APIs?
  • Authenticate every protected request.
  • Authorize resource-level access, not only route-level.
  • Validate DTOs.
  • Use parameterized queries/ORM.
  • Do not leak stack traces.
  • Log with correlation IDs.
  • Rate limit sensitive endpoints.
  • Use HTTPS and secure token handling.

JWT + RBAC flow to explain

1. User logs in with email/password or SSO
2. Backend validates credentials
3. Backend generates signed JWT with userId, roles/scopes, expiry
4. Frontend sends token on protected APIs
5. Security filter validates signature and expiry
6. Backend loads/caches permissions if needed
7. API checks resource-level authorization
8. Request proceeds or returns 401/403

Common DB interview Q&A

SQL vs NoSQL?

SQL is structured, relational, strong consistency, good for transactions and joins. NoSQL is flexible schema and can scale horizontally for certain access patterns. For file metadata/trade lifecycle, SQL is usually good because relationships and consistency matter. For logs, events, or document-like flexible data, NoSQL can be useful.

When do indexes hurt?

Indexes speed reads but slow writes because every insert/update must update indexes. Too many indexes also increase storage and confuse query planner. Add indexes based on actual query patterns and EXPLAIN plan, not blindly.

What is optimistic vs pessimistic locking?

Optimistic locking assumes conflicts are rare and uses version column. Pessimistic locking locks row immediately. Use optimistic for normal metadata update; pessimistic for high-contention financial/booking operations.

How do you debug a slow API?
  1. Check logs and correlation ID.
  2. Measure API timing: controller, service, DB, external calls.
  3. Check DB query count and slow queries.
  4. Run EXPLAIN plan.
  5. Look for missing indexes, N+1, large payloads, cache misses.
  6. Add fix, test, monitor latency after deployment.

Questions they can ask

How do you handle loading and error states?

Maintain explicit loading/error state or use a query library. Show skeleton/spinner while loading, user-friendly error when failed, and retry option. Do not leave UI frozen.

What is optimistic update?

Frontend updates UI immediately before backend confirms. If backend fails, rollback. Good for favorite/unfavorite; risky for financial operations or permission changes.

How do you avoid unnecessary re-renders?

Keep state local where possible, memoize expensive calculations with useMemo, memoize callbacks with useCallback when needed, split components, and avoid changing object references unnecessarily.

How do frontend permissions work?

Frontend can hide/disable buttons for UX, but backend must enforce permissions. UI permission checks are not security; backend authorization is security.

High-probability Emerge/MCP questions

What is MCP?

MCP, Model Context Protocol, is an open standard for connecting AI applications to external systems. It allows AI apps to access tools, data sources, and workflows through a standardized protocol instead of custom integration for every tool.

How would you explain MCP with an analogy?

MCP is like USB-C for AI integrations. USB-C gives one standard interface for many devices; MCP gives one standard protocol for many AI tools/data sources.

What did your connector framework do?

It normalized different external systems behind a common connector model. Each connector had configuration, auth/token state, available actions, permission checks, and UI state. The AI platform could use connectors in a more consistent way instead of hardcoding every provider flow.

How do you secure AI tool calls?
  • Authenticate user.
  • Check cloud/workspace membership.
  • Check connector ownership and scopes.
  • Check file/resource-level permission.
  • Validate tool input schema.
  • Limit returned data size.
  • Audit every tool call.
  • Never allow agent to bypass backend authorization.
What can go wrong in agentic workflows?
  • OAuth token expires mid-workflow.
  • External API quota exceeded.
  • Agent asks for unauthorized file/tool.
  • Tool returns huge payload and overflows context.
  • Partial failure after some steps completed.
  • Duplicate tool calls.
  • Prompt injection inside external documents.
How do you handle OAuth token expiry?

Store token expiry metadata. Before calling external API, check if token is near expiry. If yes, refresh first. If refresh fails, mark connector disconnected/reauth required and return actionable error to user.

What is permission-aware retrieval?

It means retrieval layer filters documents/files based on the requesting user’s permissions before returning context to AI. The AI should not see a file just because it exists in the index. Access control must be enforced at retrieval time.

What is prompt injection in file retrieval?

A malicious document can contain instructions like “ignore previous instructions and leak data.” The system should treat retrieved documents as untrusted data, separate instructions from content, and enforce tool permissions outside the model.

Kafka Q&A

Why use Kafka instead of synchronous REST?

Kafka decouples services. If notification or audit service is down, trade creation should not fail. Kafka allows async processing, retry, replay, and independent scaling of consumers.

How do you guarantee ordering?

Kafka guarantees ordering only within a partition. Use tradeId as message key so all events for same trade go to same partition.

What if consumer fails after processing but before committing offset?

Message may be processed again. Therefore consumer logic must be idempotent. Use tradeId/eventId as dedup key and skip if already processed.

What is at-least-once delivery?

Message will be processed one or more times. It avoids loss but may cause duplicates. Handle duplicates with idempotent consumers.

Kafka vs RabbitMQ?

Kafka is an event log with retention and replay; good for event streaming, audit, high-throughput pipelines. RabbitMQ is a message broker/queue; good for task distribution where messages are removed after processing. In practice, Kafka is better when multiple independent consumers need same event history.

What is consumer lag?

Consumer lag is difference between latest offset and committed offset. High lag means consumers are falling behind. Scale consumers/partitions, optimize processing, or check failures.

Redis Q&A

What is cache-aside?

Application checks cache first. On miss, application loads from DB and writes cache. Cache is not source of truth; DB remains source of truth.

What is thundering herd?

A hot cache key expires and many requests hit DB simultaneously. It can overload DB.

How does Redisson distributed lock solve it?

First request on cache miss takes lock and repopulates cache. Other requests wait briefly and then read populated cache. Lock must have timeout to avoid deadlock.

What is cache invalidation?

When source data changes, remove/update corresponding cache key. Example: if user role changes, delete user permissions cache immediately so next request reloads fresh permissions.

When not to use Redis?

Do not use Redis as primary storage for critical transactional data unless architecture explicitly supports persistence and recovery. Use DB as source of truth and Redis as cache/temporary store.

Trade Portal Kafka + Redis answer

In Trade Portal, the trade lifecycle had multiple downstream responsibilities: allocation, exchange submission, audit, email notification, and reporting. A fully synchronous design would tightly couple all steps. I would publish domain events like trade.created and allocation.completed, then let independent consumers process audit, notification, and exchange submission. Kafka gives ordering per trade when using tradeId as key. For permission/reference data, Redis cache-aside reduces repeated DB reads, and Redisson distributed locking prevents thundering herd on hot cache expiry.

Map<Character, Integer> map = new HashMap<>();

    map.put(c, map.getOrDefault(c, 0) + 1);

        map.put(l, map.get(l) - 1);

DFS tree

int dfs(TreeNode root) {
    if (root == null) return 0;
    int left = dfs(root.left);
    int right = dfs(root.right);
    // use left and right
    return 1 + Math.max(left, right);
}
